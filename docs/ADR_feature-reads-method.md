# 【ADR】入力読み上げ方式の比較と採用方針

## Status

## Context
このプロトタイプの入力画面には、読み上げの仕方が2つある（`InputSpeechReadMode` で切り替え）。

1. **漢字変換が確定したあとに読むやつ**（`readsConfirmedText`）  
   確定した文字列が変わったら、少し待ってからまとめて読む（`AutoReadDebounceFeature`）。読み終わったらテキストを消す想定。

2. **変換中の1文字ずつ読むやつ**（`readsCharacterByCharacter`）  
   まだ変換してる最中に、文字が増えたり減ったりするたびに読む（`CharByCharReadFeature`）。

あと、変換中はキーを打った音みたいな**効果音**も鳴る（`InputFeedbackSoundFeature`）。

読み上げは iOS 標準の **`AVSpeechSynthesizer`** を使っていて、ここはだいたい **前から順に並んで再生**される（キューに溜まるイメージ）。

調べた結果、だいたいこういうことが分かった。

* 1文字ずつ読む方は、**待ち行列に並んでる時間**も「遅れ」に入りやすい。論文とかで出てくる「1秒に何文字くらい」みたいなペースと比べると合わせにくい。
* 効果音と読み上げが同じタイミングで鳴ると、**耳がごちゃっとする**。だから今はモードを見て効果音を止める、みたいなことをしている。
* 確定してから読む方は、**読み上げの回数が少なくなりやすい**ので、待ち行列がすごく詰まりにくい。

## Decision

### 比較対象の整理

| 名前 | 中身 | 効果音との関係（今の方針） |
|------|------|---------------------------|
| **方式A** | 確定読み＋デバウンス＋効果音（`readsConfirmedText` など） | 変換中は効果音。確定したら読み上げ。役割が分かれやすい。 |
| **方式B** | 1文字ずつ読み（`readsCharacterByCharacter` など） | 読み上げが変換中にも鳴るので、効果音はモードで止める必要がある。 |

### 比較ポイント

* **読み上げの回数と待ち行列**  
  * 方式A：確定してから＆少し待つので、**鳴らす回数が少なめ**。行列に並んで待つ感じが出にくい。  
  * 方式B：打つたびに鳴らすので、**回数が多い**。行列の待ちが「遅れ」に混ざりやすい。

* **耳への負担**  
  * 方式A：短い音と、長めの読み上げが分かれやすい。  
  * 方式B：一音同士の間隔が長いので、認知負荷が重くなる懸念がある。

* **コードのわかりやすさ**  
  * 方式A：今の MVVM と Feature の分け方と合いやすい。  
  * 方式B：モードで効果音を止めるとか、細かい話が増えやすい。

* **体感の速さ・聞き取りやすさ**  
  * 方式A：ちょっと間をあけてからまとめて読むので、チャットっぽく聞こえやすいことが多い。  
  * 方式B：キーにぴったりついてくる反面、標準の読み上げのままだと**速さの目標に寄せにくい**。

### 採用するのはどっちか

#### 採用：方式A（確定読み＋効果音）

このプロトタイプでは、**基本は方式Aでいく**ことにする。

理由はこんな感じ。

* 待ち行列のせいで遅く感じにくい（読む回数が少ないから）。
* 「効果音は操作の返事」「読み上げは文章の確認」みたいに**役割が分かれやすい**。
* 方式Bより、モードと効果音の兼ね合いが**単純**になりやすい。

#### 不採用：方式Bを「メインのおすすめ」にはしない

**デフォルトのおすすめや、いちばん大事にするUXは方式Bにしない。**

理由。

* 読み上げが多すぎて、キューが溜まりやすい。
* 効果音とぶつかりやすい。設定や計測がややこしくなりやすい。
* 「打った文字を全部音声で復唱する」が目的じゃないなら、標準の読み上げだけで頑張るのは**ちょっと無理がある**。

#### 方式Bは完全に捨てるわけじゃない

* この ADR で「コードを消せ」とは書かない。**試したり比べたりする用**としてモードは残してもいい。
* でも、アプリを初めて使う人向けの**おすすめ**を決めるなら、**方式Aをデフォルト**にした方がいいと思う。

### あとから「やっぱりこうしよう」となった案（メモ）

* 方式Bで効果音もずっと鳴らす → **耳がごちゃつく**ので、メイン案にはしない。
* 方式Aだけど、確定するたびにすぐ読む（待ち時間なし）→ 読み上げが連打されて、また行列が詰まりやすい。**今みたいに少し待つ**方がいい。
* 方式Bで、いちいち止めて最新1文字だけ読む（`stopSpeaking` ばかり）→ **途中の文字が読まれない**ので、「1文字ずつ」という意味と合わない。メイン案にはしない。

## Consequences

### いいところ
* 「おすすめは確定読み＋効果音です」と説明しやすい。
* 実装の話やレビューのときに、**方針がぶれにくい**。

### わるいところ・注意
* 方式Bを残すなら、ドキュメントに**補助モード**みたいに書いておかないと、あとで混乱しそう。
* 将来、読み上げエンジンを変えたり（ストリーミングとか）、待ち行列の扱いが変わったら、この ADR の話は**もう一度見直した方がいい**。

## Compliance
* 効果音と読み上げを、同じタイミングで増やすような変更は、**耳に負担がないか**と**モードの兼ね合い**を確認してからマージした方がいい。
* この文章はあくまで**プロトタイプでどっちを推すか**の話。コードから `InputSpeechReadMode` を消せ、とは書いていない。ちゃんと決めたくなったら Status を accepted にする、くらいでよい。
