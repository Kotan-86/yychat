## Context
* 音声認識アプリ YY文字起こし と YYProbe では、チャット入力機能とチャット音声読み上げ機能が搭載されている
  * YY文字起こし と YYProbe は、Swiftフレームワークで開発されている
* それにより、ろう・難聴者の人は自身の話したいことを文字に起こして相手に伝えることができる
* しかし、以下2点の課題がある
  * チャット入力で、ろう・難聴者は「聴者を待たせる」、聴者は「チャット入力を待つ」という時間と手間、心理的障壁を排除できない
  * 音声認識による文字起こしとチャットの区別がつかず、チャットがどこにあるかが読みづらい
* そこで、一つ目の課題に対して、入力途中でも自動的に随時読み上げられる機能を開発する
* 既存の読み上げ機能は、送信ボタン押下時に文字列全体を読み上げしている。本開発で、入力中の文字列をリアルタイムで読み上げている点が技術的な相違点である

## Decision

### フロント(UI): 入力の検知

#### 採用する技術
**UIKit**
* iOSアプリ YY文字起こし と YYProbe では、UIKit を使用しており、組み込み時に整合性を担保するため

#### 比較検討
* SwiftUI: 今やモダンなフレームワークであるが、iOS15.5に対応できるバージョンでは以下の理由で不採用となる
  * YY文字起こし と YYProbe は UIKit ベースでフロント実装されており、SwiftUI と共存させると本番環境への移行時に変更範囲が多くなってしまうため
  * 入力中の文字位置特定やハイライト処理など、逐次読み上げUXに必要な精密なテキスト操作APIが不足しているため
  * 頻繁な状態更新でパフォーマンスが落ちたり、最適化のためのアーキテクチャの複雑化を避けるため

### ロジック: 読み上げの制御

#### 採用する技術
**独立したロジック**
* UIレイヤーから完全に切り離し、読み上げ処理に専念させるため
* モジュール化や SWiftUI への移行の際に、そのまま再利用できるようにするため

### エンジン: 読み上げの処理

#### 採用する技術
**AVFoundation**
* 無料かつオフラインで動作できるため
* Android には未対応だが、Androidx で同じ要件を維持して置き換えられるため
* AppleのOSに標準搭載されているため、ライブラリ容量を気にしなくてもよいため

~~**GoogleCloud TTS**~~
* ~~本番環境への移行時に、簡単に置換できるようにするため~~
* ~~iOS や Android 関係なく動作できるようにするため~~
* ~~プロトタイプ実装の速度を優先させたいため~~
* ~~ストリーミングに対応しているため~~
 
#### 比較検討
* ~~AVFoundation: AppleのOSで音声合成を含むフレームワークで完全無料・オフライン動作であるが、Android対応不可のため不採用~~
* GoogleCloud TTS: iOS, Android 両方対応できるが、クラウド型でオフラインでは動作できないため不採用
* Gemini API / OpenAI API / Azure API: 感情やイントネーションなどを表現できるが、プロトタイプ実装優先のため不採用
  * ただし、本格的に開発するとなった場合は、有力な選択肢になる
* VOICEVOX CORE: ローカルで動かせてオフライン環境でも動作できるが、ライブラリ容量が200MB~500MBと大きくなるため不採用

## Consequences

### メリット
* 既存の YY文字起こし や YYProbe などへの組み込みがしやすい
* ストリーミングかつ低レイテンシのため、チャット入力がなされるとすぐに音声が読み上げられ、スムーズなコミュニケーションにつながる？（仮説）

### デメリット・トレードオフ
* ~~GoogleCloud TTS はクラウド型でオフラインでは動作できないため、オフラインで動かしたい場合はローカルで動く音声合成エンジンの導入が必要になる~~
* Android では、Androidxに置き換える必要がある
* より自然な音声発話のニーズが高い場合は、Gemini API / OpenAI API / Azure API などの AI TTS への置換が必要になる

## Compliance
* フロントとロジック、エンジンは疎結合で実装すること
  * 将来的に、本番環境への適用や他フレームワーク移行、エンジンの置換などに対応しやすくするため
